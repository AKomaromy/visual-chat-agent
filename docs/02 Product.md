# 02 — Product

**Status:** Implementation-ready product specification, synthesized from the Deep Research corpus's two PRD tracks (`product.md`, `implementation-guide.md`) and reconciled with `01 Vision.md`.
**Companion documents:** `01 Vision.md`, `03 UX.md`, `06 Data.md`, `08 MVP.md`

---

## 1. Product summary

Mirror turns questions about a changing world into interactive visual workspaces personalized to the user's goals, projects, and expertise. It supports two core intents:

1. **Daily impact briefing** — *"What should I know today?"*
2. **Topic investigation** — *"What's happening with AI regulation?"*

The output is a coordinated workspace — Impact Radar, Timeline, Relationship Graph, conditional Map — that jointly answer the question. It is not a prose answer with charts attached.

---

## 2. Scope decision

### Build (hackathon)
Paste-profile + manual profile onboarding; editable Mirror Model graph; GDELT-primary event ingestion; entity/topic/location normalization; semantic + deterministic relevance ranking; daily briefing; topic investigation; Impact Radar; timeline; relationship graph; conditional map; coordinated filtering and cross-highlighting; evidence/provenance drawer; streamed agent progress; relevance feedback; saved workspace snapshot; provenance/pipeline mode; ClickHouse analytics; Trigger.dev orchestration; optional Postgres + CDC (gated, see `05 Architecture.md` ADR-006).

### Do not build during the hackathon
LinkedIn OAuth/scraping; broad website crawling; private GitHub repo access; team collaboration/permissions; arbitrary user-uploaded datasets; arbitrary chart generation; autonomous notifications; native mobile; enterprise SSO; billing; multi-language localization; full regulatory-document parsing across jurisdictions.

This cut protects Technical Implementation (20%) and Presentation (5%) by favoring a complete product over a broad prototype. Full P0/P1/P2 line-item scope lives in `08 MVP.md`; this document specifies *what* and *why*, not the cut line.

## 3. Target personas

### Persona A — Product/technology leader (design center)
Tracks AI, platforms, regulation, competitors, and customer behavior across several initiatives. Pain: important developments are scattered across news, newsletters, and social feeds; generic summaries don't explain impact on current priorities. Success: understands the five developments that may change product decisions, and why.

### Persona B — Founder or independent strategist
Operates with limited research support; must connect market, policy, technology, and competitor signals. Pain: research tools generate reports that take time to read and go stale. Success: a reusable visual model that can be explored and shared.

### Persona C — Analyst or innovation lead
Monitors a defined topic and must explain it to others. Pain: monitoring tools surface articles, not the structure among entities, geographies, and trends. Success: converts a complex topic into a map, timeline, and relationship graph with evidence.

The MVP is designed around **Persona A**; this creates a credible beachhead and a clean demo.

## 4. Core user stories

**Onboarding**
- I can paste a profile generated by another assistant so I don't start from a blank slate.
- I can see the extracted profile before it affects recommendations, and adjust importance, remove incorrect items, or add missing goals.
- I can begin with a manual description when I have no exported profile.
- I can see suggested questions based on my profile.

**Querying**
- I can ask what I should know today, or investigate a topic/entity/geography/date range.
- I can see processing stages while the answer is built.
- I can refine the query without losing the workspace.

**Exploration**
- I can select a signal and see every connected entity, goal, and source.
- I can filter by time, geography, topic, and relevance.
- I can hover/focus one view and see it highlighted in all others.
- I can distinguish new events from repeated coverage, see why a development ranked for me, and open the underlying evidence.

**Learning**
- I can mark a signal useful / not useful / already known, and correct the profile connection that caused a bad recommendation.
- I can observe changed rankings after feedback.

**Reuse**
- I can save a workspace and reopen the exact snapshot later. (Read-only sharing is P1.)

## 5. Primary experience

### 5.1 Onboarding flow

1. **Choose context source** — P0: paste an AI-generated profile, or describe yourself manually. P1: import a public GitHub profile by username. Deferred: LinkedIn, personal website, private connectors. *(LinkedIn integration carries API/compliance risk disproportionate to what it teaches; a pasted profile demonstrates the same personalization idea for a fraction of the build cost.)*
2. **Extract structured context** — Mirror identifies expertise, active projects, goals, organizations, industries, technologies, locations, followed people/entities, and preferred depth/cadence. Each item carries source, confidence, importance, last-confirmed time, and an editable description.
3. **Review the Mirror Model** — an interactive graph: user at center; first ring = goals/active projects; second ring = expertise/entities/locations/interests; edge thickness = importance/confidence; source badge = manual/imported/inferred. Controls: raise/lower importance, pin, remove, merge duplicates, add missing item.
4. **Starter prompts** — generate exactly two profile-grounded prompts (one briefing, one topic investigation). Never present more than four.

### 5.2 Query flow

1. **Intent and scope** — classify briefing vs. investigation; resolve time range, geography, explicit topics/entities, requested comparison, and whether a map is justified.
2. **Candidate retrieval** — hybrid retrieval (full-text + vector + topic/entity filters + recency + geographic bounds + profile connections) over ClickHouse. See `06 Data.md` §10.
3. **Signal analysis** — for each candidate: deduplicate related coverage, identify the event cluster, compute novelty/momentum/source diversity/personal relevance, attach evidence, identify affected user goals/projects.
4. **Workspace planning** — select from the four-view grammar per the rules in `03 UX.md` and `04 Visual Language.md`. Always include Impact Radar. Include timeline for any range over six hours. Include relationship graph when ≥3 entities or ≥2 profile connections exist. Include map only when location coverage is sufficient and meaningful (§6.5). Never include a component with sparse or misleading data.
5. **Stream and render** — stream task stage, progress, preliminary signal count, completed component payloads, evidence availability, and final status. The first component renders before the full run finishes.

## 6. Visual workspace specification

Full interaction and encoding rules live in `03 UX.md` and `04 Visual Language.md`. This section defines the product-level contract each view must satisfy.

### 6.1 Layout

Desktop default: **top bar** (query, time range, freshness, save, provenance) · **left rail** (Impact Radar) · **center** (relationship graph or map, whichever is primary for the question) · **bottom** (timeline) · **right drawer** (evidence + "why this matters"). Responsive fallback: tabs for Radar / Connections / Timeline / Map with a persistent selected-signal strip.

Visual hierarchy: highest-impact signal → overall pattern → supporting signals → evidence/methodology. Default visible text: title, one-sentence verdict, concise labels, one-sentence relevance explanation — nothing more (see the 80-word first-viewport budget in `03 UX.md`).

### 6.2 Impact Radar

A ranked list/radial plot of 5–7 signals. Each shows title, relevance score, direction (rising/stable/declining), novelty, time, geography badge, confidence, and top profile connection. Selecting a signal filters every view; sortable by relevance/momentum/novelty/recency; supports hide-known and useful/not-useful feedback.

### 6.3 Timeline

Shows event-cluster volume, source volume, momentum, milestones, and selected-signal emphasis. Default aggregation: hourly (≤48h), daily (49h–90d), weekly (>90d). Supports brush-to-narrow, spike inspection, and period-over-period comparison.

### 6.4 Relationship graph

Node types: developments, people, organizations, technologies, policies, locations, user goals, user projects. Edge types: affects, announced-by, regulates, competes-with, located-in, connected-to-goal, connected-to-project, corroborates, contradicts. User-context nodes use a distinct shape; event nodes scale by impact; entity nodes scale by centrality; weak edges disappear at default zoom; max 30 initially visible nodes with progressive expansion. This view is the clearest expression of the product's originality: external change visibly connected to personal context.

### 6.5 Map

Rendered only when geography is explanatory. Modes: proportional point clusters, country/region heatmap, event arcs, synchronized selected-event locations. **Acceptance rule:** ≥60% of displayed signals must have reliable geography, location must refer to the event (not the publisher), and uncertain geography is marked and excluded from aggregation.

### 6.6 Evidence drawer

Per signal: 2–5 sources where available, publisher/timestamp, corroboration count, extraction confidence, personal relevance factors, freshness, contradictory evidence, and a generated explanation capped at 60 words.

### 6.7 Provenance / pipeline mode

A compact panel (hidden by default, shown in the demo) exposing: Trigger.dev run stages and duration, retries/failures, ClickHouse rows scanned, query latency, candidate/cluster counts, data refresh time, materialized-view/index path used, model calls and cost estimate. This is the single highest-leverage feature for the 25% required-stack criterion.

## 7. Personalization model

### 7.1 Mirror Model object types

| Type | Examples | Ranking use |
|---|---|---|
| Goal | Launch AI feature; grow consulting practice | Highest-value connection |
| Project | Product rebuild; market-entry study | Active relevance |
| Expertise | Product management; GIS; AI | Semantic fit |
| Organization | Employer; client; competitor | Entity match |
| Technology | ClickHouse; LLM agents | Topic match |
| Industry | Analytics; retail; public policy | Domain relevance |
| Location | Toronto; Canada; EU | Geographic relevance |
| Person/entity | Policymaker; founder; company | Direct mention |
| Preference | Strategic depth; daily cadence | Output shaping |

### 7.2 Relevance score

A deterministic weighted score computed **after** candidate retrieval — never a model-invented number.

| Factor | Weight | Meaning |
|---|---:|---|
| Semantic profile match | 30% | Similarity to goals, projects, expertise |
| Explicit entity/topic match | 20% | Direct match to pinned interests |
| Goal or project impact | 15% | Model-classified relationship, verified against profile |
| Novelty | 10% | Difference from previously shown clusters |
| Momentum | 10% | Change in coverage/event activity |
| Geographic relevance | 5% | Connection to priority locations |
| Source credibility/diversity | 10% | Evidence quality and corroboration |

Adjustments: "already known" feedback reduces novelty for that cluster; "not useful" reduces the responsible profile-edge weight; pinned topics get a bounded boost, never absolute priority; extreme urgency may override weak profile match but must be labeled a broad-impact alert; **no model-generated number may appear unless it traces to stored factors.**

### 7.3 Explainability

Every recommendation must answer: which profile items contributed, which evidence caused the score, was it included for urgency despite weak relevance, and what feedback would change future ranking.

## 8. Data sources

### 8.1 P0 source: GDELT (primary — see `01 Vision.md` §7 for why this reverses one research track's RSS-first recommendation)

GDELT provides near-real-time global event and media metadata, entities, organizations, locations, themes, multilingual coverage, and event/relationship structure — a natural fit for maps, timelines, and graphs, and it removes the highest-risk custom extraction work from the build. Ingest a bounded rolling window, not the full historical corpus.

Recommended demo window: previous 7 days, selected AI/technology/policy/business topics, enough history to compute novelty and momentum.

### 8.2 P1 sources
Curated publisher RSS/Atom (source diversity and clean attribution); U.S. Federal Register API; EUR-Lex/selected EU policy feeds; official company newsroom RSS; GitHub public API for onboarding; user-provided source URLs.

### 8.3 Source policy
Preserve canonical URL, publisher, timestamp, retrieval time. Never present an LLM summary without its supporting records. Deduplicate syndicated articles. Prefer original regulatory/company sources when available. Mark GDELT-derived metadata as *media-observed*, not ground truth. Store source terms/licensing notes; metadata-and-excerpt only, never unlicensed full text.

## 10. Non-functional requirements

**Performance:** first progress state < 500 ms; first meaningful component < 10 s; complete workspace < 30 s on the demo scenario; cached repeat query < 5 s; local visual filter response < 100 ms; server-backed filter response target < 1 s; no UI blocking during background tasks. (Full performance budget table lives in `08 MVP.md`.)

**Reliability:** idempotent ingestion; duplicate-safe event processing; resumable chat stream; bounded-backoff retry; component-level degradation; clear failed-state messaging; preloaded demo-data fallback.

**Security and privacy:** tenant-isolated user context; no secrets in the public repository; server-side API credentials; least-privilege database credentials; explicit consent before external-profile import; deletion control for profile data; source text stored only as permitted; no personal profile in public demo fixtures without permission. Full model in `05 Architecture.md` §11 and `06 Data.md` §20.

**Accessibility:** keyboard navigation; visible focus states; non-color indicators for importance/status; text alternative for every chart; minimum contrast; keyboard-reachable tooltips; reduced-motion mode. Full checklist in `04 Visual Language.md` §17.

**Observability:** every query tied to a workspace and a Trigger.dev run; per-stage duration; database query latency; model usage; failure reason; ingestion freshness; source count; user feedback.

## 11. Error and edge states

| Condition | Product behavior |
|---|---|
| Empty or vague profile | Ask the user to confirm three goals; use broad signals temporarily |
| No relevant developments | Show "No material change detected" plus baseline context — never filler |
| One dominant source | Lower confidence and surface source concentration |
| Conflicting sources | Contradiction badge; separate evidence groups |
| Missing geography | Omit map; explain in the component selector |
| Slow enrichment | Render Radar and timeline first; add graph later |
| Source API unavailable | Use stored rolling window; show data cutoff |
| Model output invalid | Retry workspace planning once, then fall back to a deterministic default layout |
| Unrelated general question | Explain Mirror's supported scope; offer a visual investigation prompt |
| Oversized result | Limit to top clusters; allow explicit expansion |
| Sensitive profile detail | Exclude from ranking unless the user explicitly pins it |

## 12. Analytics and success metrics

**Activation:** onboarding completion; time to first query; time to first useful signal; % correcting ≥1 profile item.
**Relevance:** useful/not-useful ratio; top-5 precision from user rating; "already known" rate; correction rate by profile item; source-open rate.
**Engagement:** component interactions per workspace; filter use; graph expansion; save rate; return usage; follow-up rate.
**Trust:** evidence-drawer opens; contradiction interactions; confidence complaints; unsupported-claim rate; source coverage.
**System:** ingestion lag; first-component latency; completion latency; ClickHouse query latency; Trigger.dev retry rate; component failure rate; cost per workspace.

**Hackathon dashboard** (shown in the demo): total events ingested, latest ingestion time, sources represented, ClickHouse query latency, active/completed Trigger runs, workspace completion time, component generation time, feedback events.

## 13. MVP acceptance test

A new user must complete the following without explanation:

1. Paste a provided profile.
2. Review and edit one goal.
3. Ask "What should I know today?"
4. See visible processing stages.
5. Receive a ranked visual workspace.
6. Select a signal.
7. Observe coordinated updates across timeline, graph, and map.
8. Inspect source evidence.
9. See why the signal relates to the edited goal.
10. Mark the signal not useful.
11. Rerun/refresh ranking and observe a change.
12. Save and reopen the workspace.
13. Open provenance mode and see ClickHouse and Trigger.dev involvement.

**Pass criteria:** no dead ends; no uncited signal; no text wall; no component with misleading empty data; no manual database intervention; no exposed secret; demo completes in under five minutes.
