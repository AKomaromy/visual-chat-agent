# 03 — UX

**Status:** Engineering-ready interaction specification, synthesized and trimmed to hackathon scope from the Deep Research corpus's `dashboard-patterns.md`, `interaction-model.md`, and `response-composer.md`.
**Companion documents:** `02 Product.md` (workspace layout/regions), `04 Visual Language.md` (encoding/visual rules)

This document governs *how* the user asks, sees, inspects, manipulates, compares, verifies, and continues a conversation with a visual response. `04 Visual Language.md` governs *what* each view looks like. `02 Product.md` §6 governs *where* things sit on screen — it is not repeated here.

---

## 1. Interaction thesis

Natural language expresses intent. Visual interaction inspects evidence, changes scope, compares alternatives, and surfaces follow-up questions. Mirror's division of labor:

**Ask** with language → **See** the answer as a visual workspace → **Inspect** with focus/selection → **Change** scope via direct manipulation → **Compare** by pinning → **Verify** via evidence/provenance → **Continue** with a follow-up that inherits visual context.

Interaction replaces explanation whenever the user can learn faster by manipulating evidence than by reading prose.

---

## 2. Four disclosure depths

Every response supports four levels of engagement. The interface must never require Verify-level reading to understand the Glance-level answer, and must never hide freshness, confidence state, or evidence access entirely.

| Depth | Question | Default behavior | Text role |
|---|---|---|---|
| **Glance** | "What matters?" | Verdict, primary view, top marks, visible scope | One concise verdict |
| **Inspect** | "What is this?" | Focus, tooltip, linked highlight, exact values | Labels and compact facts |
| **Explore** | "What else is here?" | Select, filter, brush, compare, expand, ask-about-selection | Prompts and state labels |
| **Verify** | "Can I trust this?" | Evidence, uncertainty, source, freshness, lineage, assumptions | Detailed, on demand |

**Evaluation targets** (product targets, not universal law): 3 seconds to state topic/scope/primary conclusion; 10 seconds to identify the largest change or leading signal; 30 seconds to inspect evidence or alter scope unaided; 2 minutes to form and test a new question.

## 3. Response patterns (question-family routing)

Mirror routes every question to exactly one of **four hackathon MVP response patterns**. This reconciles the broader 9-pattern task-driven design system in the source research with the tighter 4-view product grammar in `02 Product.md` — these four are the intersection both source tracks converged on as MVP-ready.

| Pattern | Best for | Primary view | Supporting views | Maps to `02 Product.md` |
|---|---|---|---|---|
| **Daily Briefing** | "What happened today?", "What should I know?" | Impact Radar (ranked event timeline) | Category deltas, evidence, map only if geography explains importance | §6.2 |
| **Topic Atlas** | "What's happening with AI regulation?", broad domain orientation | Relationship graph or topic landscape | Trend small multiples, event timeline, actor/jurisdiction comparison | §6.4 |
| **Change Lens** | "What changed this week?", explicit period comparison | Timeline in delta mode (sorted delta bars / slopegraph) | Event strip, spatial difference when geography changed | §6.3 |
| **Pattern Finder** | "What patterns am I missing?", discovery | Impact Radar in anomaly-ranked mode | Ranked pattern queue, context distribution, timeline/map/graph chosen by the pattern's structure | §6.2 + §6.4 |

Each pattern reuses the same four view *types* (Impact Radar, Timeline, Relationship Graph, Map) with a different primary view and internal mode — Mirror does not introduce new top-level view types per pattern. Decision Canvas, Comparison Board, Spatial Situation Room, Relationship Explorer, and Event Trace (present in the broader source research) are explicitly deferred past the hackathon; see `08 MVP.md`.

### 3.1 View-role rule

A view earns screen space only if it plays a unique role: **Answer** (direct result), **Context** (baseline/comparison), **Explain** (associated event/entity/relationship), **Locate** (geography), **Verify** (evidence/uncertainty), **Act** (support a next step). Two views sharing a role are redundant unless they encode materially different scales.

### 3.2 First-viewport contract

Every response contains, within the first viewport: one verdict, one dominant view, zero to three supporting views, visible scope/freshness, one obvious next action. **Never more than four analytic objects above the fold.**

## 4. Core interaction grammar

| Gesture | Purpose | Key rule |
|---|---|---|
| **Preview** (hover/keyboard focus) | Inspect without committing | Highlights linked marks in other views; clears on exit; never changes scope or verdict |
| **Select** (click/tap/Enter) | Establish a persistent analytical subject | Strong emphasis across all views + removable selection chip; enables "ask about selection"; does **not** auto-filter |
| **Filter** (include/exclude) | Change the population used to compute the answer | Updates scope bar and result count; recomputes affected views/verdict; visible undo |
| **Brush** (continuous range) | Narrow time/quantitative scope | Preview while dragging when cheap; commit on release for expensive queries; adds a scope chip with exact bounds |
| **Zoom/pan** | Change scale or navigate | Navigation by default, not a filter — data scope changes only when a scope chip updates |
| **Pin** | Preserve an item/period/scenario for comparison | Stable pin identity; default cap of 2–3 pins; clear "unpin all" |
| **Ask about selection** | Bridge manipulation back into language | Places the selection + inherited scope into the composer; suggests ≤3 context-sensitive follow-ups |
| **Reset/undo/redo** | Make exploration safe | Undo every persistent selection/filter/baseline/view-mode change; reset current view separately from full response |

**Highlight vs. filter vs. recompute — these must look and behave differently:**

| Effect | Meaning | Visual treatment | Verdict behavior |
|---|---|---|---|
| Highlight | Correspondence within the same population | Accent outline; other marks retain opacity | Unchanged |
| Filter | Included population changed | Scope chip; excluded marks muted/removed | Recomputed |
| Recompute | New analytical operation ran | Updating status on affected views | Provisional, then updated |

## 5. Coordinated multiple-view behavior

### 5.1 Coordination bus

Every view participates in one shared contract: entity key, time key/granularity, geographic key/boundary level, relationship key (if applicable), active measures, focus, selection, filter state, comparison pins, baseline, provenance pointer. A view subscribes only to dimensions it understands (a map need not respond to a relationship selection with no geographic binding).

### 5.2 Update ordering

1. Initiating mark responds immediately.
2. Linked highlights update.
3. Scope/selection chips update.
4. Fast local views update.
5. Background recomputations stream as they complete.
6. Verdict updates only after supporting evidence is validated.

Do not animate every view at once; preserve object position/scale wherever possible.

### 5.3 Selection propagation

Propagate when meaning is stable: entity→entity across chart/timeline/map/evidence; interval→time-aligned views; region→regional rankings/trends; topic→related events/actors/sources; relationship→its endpoints and supporting evidence. Do not propagate when it would falsely imply identity (e.g., matching two records only by similar labels).

## 6. Filters

| Category | Examples | Default disclosure |
|---|---|---|
| Scope | time, geography, population | Always visible |
| Subject | topic, entity, event type | Visible when set |
| Comparison | baseline, cohort, scenario | Visible when relevant |
| Personal relevance | goal, project, expertise | Visible when materially shaping ranking |
| Evidence | source type, confidence, quality | Contextual/advanced |
| Analytical | aggregation, normalization | Advanced |

Rules: active filters are compact, removable chips; a collapsed "more filters" shows the active count; single-value filters may apply immediately, multi-value/expensive ones use explicit Apply/Cancel; exclusion is visibly distinct from inclusion; reset restores the last meaningful state, not always the original query; filters persist across compatible follow-ups; disabled values show why they're unavailable.

---

## 7. Compare mode

A first-class state, not a separate dashboard. Entry: pin two entities, choose "Compare" from a selected mark, ask a comparative follow-up, select a baseline, or brush two time windows.

Layout by task: **values** → aligned dot plot/bars; **before/after** → slopegraph/dumbbell; **trends** → synchronized lines/small multiples; **geography** → synchronized maps + delta ranking; **networks** → neighborhood comparison/matrix difference.

Rules: shared scales; explicit reference item; sort by meaningful difference; distinguish absolute vs. relative difference; preserve sample size/uncertainty; "show only differences" always has a route back to the full profile. Exiting compare mode returns to the prior selected state.

---

## 8. Progressive disclosure of controls

- **Always visible:** scope/freshness, verdict, primary interaction affordance, selection state, evidence access, reset/undo after a state change.
- **Contextual:** compare after selection, filters relevant to the active view, graph neighborhood expansion, alternate baseline, uncertainty detail, "ask about selection."
- **Advanced:** normalization method, confidence threshold, source weighting, aggregation resolution, graph layout/relationship type, diagnostic query/job lineage.

A control appears when it can answer the next likely question — never merely because the data model happens to support it.

---

## 9. Discoverability

Essential actions use visible labels, not icons alone. Interactive marks respond to hover/focus with a consistent cue. The first response may show **one** lightweight coach mark ("Select a change to inspect it") and never repeats it once learned. Show no more than three suggested actions, generated from current state (e.g., "compare the top change," "narrow to the selected region," "show the source evidence") — never generic ("Explore more").

---

## 10. Latency and job feedback

| Expected duration | Behavior |
|---:|---|
| < 100 ms | Update directly; no progress indicator |
| 100–500 ms | Immediate active-state feedback |
| 500 ms – 2 s | Preserve old result; show localized progress |
| > 2 s | Show named stages; stream partial results |
| Long-running enrichment | Cancellable background job; workspace stays usable |

Use concrete stage language ("Interpreting scope," "Comparing periods," "Finding related events," "Composing the visual response") — never "Thinking…" or fake precision percentages. Focus/local highlighting may update optimistically; filters that change analytical results require validated data before the verdict changes, and a provisional verdict must be labeled "updating." On scope change before a job completes: cancel obsolete work where practical, mark late results superseded, never insert a result into a newer state.

---

## 11. Evidence and verification

The evidence drawer opens from the verdict, a selected mark, or a source indicator, and includes: supporting records, exact timestamps, source type/quality, transformations/aggregations, uncertainty/missingness, assumptions, ClickHouse data lineage, Trigger.dev job lineage, and an accessible table/textual representation.

**Bidirectional linking:** selecting a source highlights the marks it supports; selecting a mark filters the evidence list; hovering the verdict highlights its supporting marks; conflicting sources stay visible and comparable; **unsupported claims cannot render as final verdicts.**

**Progressive verification layers:** (1) confidence cue in the main response → (2) evidence summary (source count, freshness, disagreement) → (3) full lineage (records, transformations, jobs).

---

## 12. Uncertainty interaction

Show intervals/distributions/confidence encoding directly when uncertainty affects interpretation; suppress visual dominance when uncertainty is high; identify low sample size or source disagreement; distinguish absence of evidence from evidence of absence. Focus reveals exact range/method; compare mode aligns uncertainty across options; "test robustness" reruns the result under alternate assumptions (source exclusion, alternate baseline, minimum sample size). Use compact, specific language ("Moderate confidence; two sources disagree") — never unexplained labels like "AI confidence 72%."

---

## 13. History

History records meaningful state changes only (original question, scope changes, baseline changes, selected/pinned items, follow-ups, saved snapshots) — never transient hover states. A shared/saved response preserves query, scope, selected pattern, active filters/pins, visible annotations, data freshness, evidence links (where permitted), and a clear live-vs-frozen indicator. Branching from an earlier state does not lose later work (deferred past MVP — see `08 MVP.md`).

---

## 14. Accessibility interaction model

- **Keyboard:** move through scope, verdict, views, marks, controls, evidence in logical order; inspect via focus; select with Enter/Space; adjust ranges with keyboard; open/close drawers without losing focus; undo, reset, compare, and ask-about-selection all keyboard-reachable; skip repetitive chart regions.
- **Screen reader:** each view exposes purpose/conclusion, axis definitions, current filters/baseline, mark/group count, selected state, a concise trend description, and a structured data table; announce scope/verdict/job-state changes.
- **Color and alternatives:** never encode status with color alone; redundant shape/position/pattern/label/icon; sufficient contrast; high-contrast and reduced-motion support; no hover-only dependency.
- **Cognitive:** consistent interaction semantics across views; plain action-oriented labels; limited concurrent choices; preserved context during drill-down; undo/reset always available.

Full visual accessibility rules (contrast, WCAG baseline) live in `04 Visual Language.md` §17.

---

## 15. Error, empty, stale, and partial states

- **Partial response:** completed views stay interactive; a failed view keeps its reserved space with a concise role-specific message; retry affects only the failed stage; verdict stays provisional if a failed view was required to support it.
- **Empty result:** show the exact scope that produced no data, an empty visual frame (not a generic error page), and up to three repairs.
- **Stale data:** mark visibly, keep inspectable, offer refresh; never blend fresh and stale views without explicit labeling.
- **Contradictory data:** show disagreement rather than hiding it; allow comparison by source; block a definitive verdict when conflict is material.

---

## 16. Touch and mobile

Tap = select, not hover; a second tap or explicit action opens inspection; pinch/pan remain navigation unless scope visibly changes; brushing uses touch-sized handles. Mobile composition: verdict + primary view first; supporting views stack in analytic order; coordinated selection persists across cards; evidence moves to a full-screen sheet; dense graphs collapse to ranked lists; maps pair with a synchronized place list.

---

## 17. Hackathon MVP interaction boundary

**Required:** focus + linked highlight; persistent selection; global scope chips; one brush interaction; filter with live recomputation; compare/pin mode; "ask about selection" context inheritance; evidence drawer; progressive view streaming; undo/reset; keyboard access for the primary path; partial-view failure handling.

**Strongly preferred:** baseline switching; visible ClickHouse query freshness; visible Trigger.dev job stages; one uncertainty interaction; responsive mobile composition; reduced-motion behavior.

**Defer:** arbitrary layout editing; complex lasso operations; multiplayer cursors; unrestricted graph expansion; custom gesture systems; voice-only operation; elaborate animation timelines; conversation branching.

---

## 18. Acceptance scenarios

**"What should I know today?"** → Daily Briefing streams with exact time scope → focus on a signal highlights topic/geography/evidence → selection persists and enables "ask about selection" → a time brush recomputes supporting views and verdict → evidence opens without replacing the workspace → reset restores the original day.

**"What's happening with AI regulation?"** → Topic Atlas presents major themes → selecting a theme filters trends/events while preserving field context → pinning a second theme enters compare mode → changing the horizon updates momentum and evidence → a follow-up ("only regulation") inherits the selection.

**"What changed this week?"** → Change Lens displays the explicit baseline → selecting a delta reveals before/after values → associated events highlight without being labeled causal → switching the baseline recomputes and preserves selection → undo restores the prior comparison.

**"What patterns am I missing?"** → Pattern Finder ranks patterns by impact/confidence → selecting a pattern highlights supporting marks → "test robustness" runs alternate assumptions as a named job → the result shows whether the pattern persists → dismissing removes it from the active queue but keeps it in history.

---

## 19. Acceptance checklist

A response is ready only when: the primary view answers the dominant task; the verdict is supported by visible marks; every supporting view has a unique role; map/graph/timeline use is justified by the question; active scope and baseline are explicit; linked selection is consistent across views; the first viewport stays under four analytic objects and ~80 visible words; uncertainty and evidence are available without dominating the canvas; keyboard and touch users can complete the primary task; loading/stale/empty/partial-failure states are defined; ClickHouse and Trigger.dev contribution is visible in product behavior; the pattern produces a demonstrable insight within ten seconds.
